## 1. ゲームの全体像

### モード
- **ひとりであそぶ（VS COM）** — CPU対戦。勝ち抜き「○人抜き」モード搭載
- **ふたりであそぶ（VS 2P）** — 1枚のキーボードを共有するローカル対戦
- **ネットたいせん（ONLINE）** — 4桁の「あいことば」で繋がるWebRTC P2Pオンライン対戦

### 操作
| 操作 | 1P | 2P（ローカル） |
|---|---|---|
| 移動 | A / D | ← / → |
| ジャンプ | W | ↑ |
| ガード | S | ↓ |
| 弱攻撃 | J | , |
| 強攻撃 | K | . |
| 必殺技 | L | / |

※オンライン対戦時は両者とも1P操作。スマホはバーチャルボタン自動表示。

### 戦闘システム
- 2ラウンド先取・60秒タイマー（タイムアップ時は残り体力勝負）
- 必殺ゲージ（攻撃ヒット+14 / 被弾+9 / ガード+4〜5、100で必殺技解放）
- ガード（被ダメージ85%カット、削りダメージあり）
- ヒットストップ（弱5F/強9F）、画面揺れ、ヒットスパーク、コンボカウント、必殺技名バナー、KOスローモーション
- 必殺技は3タイプ: **projectile**（飛び道具）/ **rush**（突進）/ **area**（周囲爆発）

### 演出
- CRT風スキャンライン＋ビネット（CSS）
- チップチューンBGM/SFXをWebAudioでリアルタイム合成（タイトル96bpm / バトル150bpm、矩形波リード+三角波ベース+ノイズハット）
- MS Gothicによるビットマップ風フォント、偽コピーライト「© 19XX DOT DAMASHII PROJECT」

### ドット絵データ形式（独自）
キャラのドット絵は**パレット＋テキストグリッド**でデータ化。画像ツール不要でAIエージェントが直接「描ける」形式にしたのが最大の発明ポイント。

```js
palette: { "o": "#1a1024", "s": "#f0c8a0", ... },   // 1文字=1色、"."=透明
frames: {
  idle: [ [ "....48文字....", /* ×48行 */ ], [ /*2枚目*/ ] ],
  walk: [...], light: [...], heavy: [...], special: [...],
  hit: [...], guard: [...], jump: [...], ko: [...]   // 計13フレーム
}
```
- 実行時に `sprites.js` がCanvasへ展開し、左右反転版もキャッシュ
- 必殺技がprojectileの場合は16×16×2枚の弾スプライトも同形式で持つ
- `tools/validate.js` が行数・文字数・パレット整合・ステータス範囲を機械検証し、`tools/preview.js` でターミナル上に実際の色で描画して目視確認できる

### 戦闘エンジン
- 内部解像度640×360・60fps固定ステップ（accumulator方式）・キャラは48px素材を96pxに拡大描画
- 状態機械: idle/walk/jump/guard/light/heavy/special/hit/ko/win
- 攻撃判定: startup→active→recoveryのフレームデータ制。activeウィンドウ中に攻撃ボックス×食らいボックスの矩形判定（1技1ヒット）
- ノックバックは weight で減衰、launch属性で打ち上げ
- **決定論設計**（同じ入力列→必ず同じ結果）: 後のオンラインロックステップの土台。乱数は演出（FX）とCPU AIのみに限定し、対戦ロジックには使わない

### オンライン対戦（ネットコード）
**遅延4フレームの決定論ロックステップ方式**（格闘ゲームの古典的本格手法）。

1. 毎フレーム、自分の入力を7bitのビットマスク（左/右/跳/防/弱/強/必）にエンコードし「4フレーム未来」のフレーム番号を付けて送信
2. シミュレーションは「自分と相手、両方のフレームNの入力」が揃ったときだけ1フレーム進む
3. 両クライアントが同一の決定論エンジンを回すため、**送るのは入力だけ**（座標やHPは送らない）
4. 60フレームごとに状態ハッシュ（両者の座標・HP・ラウンド・フェーズ）を交換照合し、同期ずれを常時監視
5. **readyハンドシェイク**: バトル画面に入ったら「準備完了（エポック番号付き）」を相手に通知し、相手の準備完了を受け取るまで入力送信を保留（入力バッファ初期化の競合を構造的に排除）

通信層（net.js）:
- PeerJS（WebRTCのDataChannel・reliable設定）。ピアID `dotdama87-XXXX`（4桁数字=あいことば）、ID衝突時は自動再生成
- ping/pongで2秒ごとにRTT計測（画面にPING表示）
- 切断検知: DataChannelのcloseイベントに加え、pong7秒途絶watchdog + connectionstatechange監視の二重フォールバック（Chromiumはタブ急クローズでcloseが発火しないことがある既知問題への対策）
- マッチング仲介のみPeerJSクラウドを利用。**対戦データはP2P直結でサーバーを経由しない**

### スマホ対応（touch.js）
- タッチデバイス検出（または `?touch=1`）で自動有効化。非対応環境ではDOMを一切作らない
- ファミコン風デザインのバーチャルボタン（十字キー＋弱/強/必/防/跳＋START/もどる）をJSが自前で生成
- Pointer Events + ポインタID個別管理で**マルチタッチ対応**（移動しながら攻撃可能）
- `Input.setVirtual(code, bool)` でキーボードと完全等価に注入 → **オンライン対戦もスマホからそのまま参加可能**
- `GAME.state === "netjoin"` を監視して、部屋コード入力時だけテンキー（0-9/⌫/OK）を表示

## 7. バグと解決の記録（学びどころ）

### 🐛 バグ①: タイトルに「ネットたいせん」が表示されない
- **症状**: オンライン対戦を実装・公開したのに、タイトルメニューが2項目のまま
- **原因**: ロジック側（main.js）はメニュー3項目に更新したが、**描画側（ui.js）のメニュー文字列リストを2項目のまま更新し忘れ**。機能は存在するが見えない状態
- **見逃した理由**: E2Eテストがキー操作（S×2→Enter）で直接遷移して検証していたため、「画面に表示されているか」を見ていなかった
- **教訓**: 状態遷移のテストとUI表示のテストは別物。スクリーンショットの目視確認を省略しない

### 🐛 バグ②: オンライン対戦が「つうしん まちあわせちゅう…」で永久停止
- **症状**: 実プレイで両画面ともラウンド1のまま停止。PINGは1msで正常
- **原因**: 両者がバトル画面に入るタイミングがずれると、**後から入った側の入力バッファ初期化（clearInputs）が、先に入った側が送信済みの入力データを消してしまう**。お互いに「相手のフレームNの入力」を永遠に待つデッドロック
- **自動テストをすり抜けた理由**: テストでは両クライアントがほぼ同時刻にバトルへ入っており、競合ウィンドウが実質ゼロだった
- **修正**: **readyハンドシェイク**を追加。「自分の初期化が終わった」ことをエポック番号付きで相手に通知し、相手のreadyを受け取ってから入力送信を開始する設計に変更。タイミング競合が構造的に発生しなくなった
- **再発防止**: ゲストを1.5秒フリーズさせてズレを人工的に再現する `netplay_skew_test.py` を追加

### 🐛 小バグ: PowerShell 5.1の文字コード問題
- deploy.ps1（日本語コメント入り）が「BOM無しUTF-8」だと日本語環境のPS5.1にShift-JISとして誤読されパースエラー → **BOM付きUTF-8で保存**して解決
- 資格情報マネージャーのトークンBlobがUTF-16で入っているケースがあり、UTF-8デコードで制御文字が混入 → NUL検出でUTF-16フォールバックを実装

### 🐛 バグ③: スマホでバーチャルボタンがゲーム画面に被る（v2.1で修正）
- **症状**: 実機（2画面フォルダブルスマホの見開き状態）で、弱/強/必/跳ボタンがゲームキャンバスのセリフ表示エリアを覆い隠す。キャンバス下に大きな余白があるのにボタンがそこに置かれない
- **原因**: 初版のタッチUIはビューポート基準（vmin/vh）の固定配置で、「画面下端に置けばキャンバスに被らない」という横長画面前提の暗黙の仮定があった。フォルダブル見開き（ほぼ正方形 0.86:1）や縦長画面では、中央寄せされたキャンバスと画面下端の間にボタンが食い込む
- **修正（v2.1）**: 「縦か横か」のorientation分岐ではなく、**`#frame.getBoundingClientRect()` で実測した「フレーム下の空きスペース」ベースの判定**に変更
  - 空きが十分 →「**下置きモード**」: ゲーム画面を上寄せし、ボタンを完全にキャンバス外の下空間へ（ボタンは従来より大型化、見開きで110px）
  - 空き不足（横持ち）→「**オーバーレイモード**」: 四隅下端ギリギリ・縮小・半透明（opacity 0.55）で被り最小化
  - resize/orientationchange＋300msポーリングで再計算（フォルダブルの開閉による動的な画面サイズ変化に追従）
- **再発防止**: 縦長(412×915)・正方形(860×1000)・横長(915×412)の3画面パターンで「全ボタン矩形×フレーム矩形の幾何学的非重複」をassertするテストを追加（touch_test.py 計40項目）
- **教訓**: 「実機のスクショ1枚」がテスト40項目分の価値を持つ。レイアウトはデバイスの多様性（フォルダブル等）を前提に、固定値でなく実測ベースで組む

### 🔧 デプロイの工夫: gh CLIが無い環境でのGitHub Pages公開
- このマシンにはGitHub CLI（gh）が未インストールだったが、Windows資格情報マネージャーに `git:https://github.com` のGCMトークン（repoスコープ）が保存されていた
- **`tools/deploy.ps1`**: Win32 CredRead API（P/Invoke）でトークンを取得し、GitHub REST APIを直接呼んで「リポジトリ作成→push→Pages有効化→公開確認」まで自動化。トークンは画面に一切表示しない設計

## 10. 学び・ふりかえり

1. **「AIが直接描けるアセット形式」の設計が突破口**: ドット絵をテキストグリッド化したことで、画像生成ツールなしにサブエージェントがキャラを丸ごと制作できた。検証（validate）とプレビュー（preview）をセットで用意したことで品質も担保
2. **決定論エンジンを最初から意識すると、後からオンライン化できる**: 乱数を演出とAIに隔離していたため、ロックステップ移植はエンジン改修ほぼゼロ（入力注入の口を開けただけ）で済んだ
3. **並列サブエージェントは「仕様書・検証ツール・ファイル所有権・自己レビュー義務」の4点セットで機能する**: 11体使ってコンフリクトゼロ、手戻りほぼゼロ
4. **テストが通っても「目で見る」工程は削れない**: 表示漏れバグもタイミングバグも、実プレイ・実画面で初めて顕在化した。自動テスト＋スクショ目視＋実機プレイの三段構え
5. **タイミング系のバグは「揺らぎを人工的に作るテスト」で再発防止する**: スキューテストのように、運次第だった競合条件をテストで強制再現できる形にしておく
