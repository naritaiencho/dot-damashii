## 1. v1プロトタイプの実測トークン（確定値）

トランスクリプト（`*.jsonl`）を集計スクリプトで解析した実数。

### メインエージェント（v1完成まで・APIコール97回）
| 種別 | 量 | コストの重み |
|---|---|---|
| **出力トークン** | **23.4万** | 最も高価（コード生成・思考） |
| 新規入力 | 1.6万 | 通常価格 |
| キャッシュ書込 | 66.0万 | 通常の約1.25倍 |
| キャッシュ読出 | 2,889万 | **通常の約1/10（激安）** |

### サブエージェント4体（キャラ制作）
- 1体あたり約9〜11万トークン
- 合計 **約39.4万トークン**

### v1の「実コスト帯」小計
```
出力23.4万 + 新規入力1.6万 + キャッシュ書込66万 + サブ39.4万 ≒ 約130万トークン
```
- 総処理量（キャッシュ読出込み）で言えば **約3,020万トークン**
- API従量課金換算で **おおよそ$50〜90（約1万円）相当**

### プロジェクト全体（v2＋オンライン対戦まで）
- メイン出力 **62万** ／ サブエージェント11体 約98万 ／ キャッシュ読出 **1.64億**
- 実質ベースで **約260〜300万トークン**、総処理量なら **約1.7億トークン**

## 2. 一番大事な概念：トークンには「2つの数え方」がある

| 数え方 | v1の値 | 意味 |
|---|---|---|
| **総トークン処理量** | 約3,000万 | APIが実際に処理した量。見た目はデカいが9割超はキャッシュ読出 |
| **実質的な消費量** | 約130万 | 新規に生成・処理された分。プラン制限に効くのは概ねこちら |

➡ 人に説明するときは「**実質130万トークン規模（総処理量ならキャッシュ込み3,000万）**」と両方言うと正確で誤解がない。

## 3. プロンプトキャッシュの正体（コスト最適化の核心）

### 仕組み
- AIは**毎回の応答で会話履歴の全体を読み直す**（ステートレス）。
- 同じ履歴の再読み込みは「プロンプトキャッシュ」が効き、**通常入力の約1/10のコスト**になる。
- Claude Codeはこれを自動でやってくれるので「長いセッションで作り込む」スタイルが成立する。

### ⚠️ キャッシュの有効期限は約5分
ここが今回ハマったポイント。
```
前の応答から5分以上空く → キャッシュ失効 → 数十万トークンの履歴をフルプライス+αで再登録
```
- つまり「考えながらポツポツ質問する軽い雑談タイム」が、実は**一番割高**になり得る。
- 実測でもセッション後半のキャッシュ再登録は**358万トークン**に達していた。

### 実体験：26% → 51% 事件
- ゲーム開発が一段落した後、「Proでも作れる？」など**短い質問を数回**しただけで、5時間制限が26%→51%に急上昇した。
- 原因の二段重ね：
  1. このセッションの文脈量が数十万トークン級まで膨張していた（ゲーム開発全史を引きずっている）
  2. 質問の間隔が5分以上空き、毎回キャッシュが切れて履歴をフル再登録していた
- 「ちょろっとした質問」でも、重い文脈を抱えたセッションでは全然ちょろくない。

---

## 4. 5時間制限は「ローリングウィンドウ」

- 5時間制限は**直近5時間の使用量**。時間が経つと古い消費は枠から外れていく（累積ではない）。
- だから「26%でゲーム1本作れた」わけではなく、**序盤〜中盤の大量消費はもうリセット済み**で、残っているのは直近分だけ。
- 開発全体の実感に近いのは **7日間制限（7d）** の方。ゴリゴリ使った痕跡はそちらに出る。

## 5. 実践ルール（今後の運用）

### ✅ セッションを用途で分ける（最重要）
| 用途 | やり方 | 理由 |
|---|---|---|
| 重い開発の続き | 同じセッション | 文脈（コード・設計）が必要 |
| 一般質問・雑談 | **新しいセッションを開く** | 文脈ゼロから始まるので激安。新セッションなら例の質問群は合計1%も使わなかったはず |

### ✅ 長い開発が一段落したら `/compact`
- 履歴を要約して圧縮 → 以降の毎回の再読み込みが軽くなる。

### ✅ ステータスラインで消費を可視化
- `Ctx`（コンテキスト使用率）・`5h`・`7d` を常時表示するように設定済み。
- こういう急上昇に**その場で気づける**のが大きい。

## 6. プラン別の見立て（※概算・要注意）

### モデルについての訂正（重要な学び）
- 当初「ProだとSonnet中心の開発になる」と説明したが、**これは誤り**だった。
- **2026年6月時点では、Claude Fable 5はPro / Max / Team / Enterprise全プランで追加費用なしで利用可能**（ローンチ〜**2026年6月22日**の提供ウィンドウ。以降は使用クレジットが必要になる見込み）。
- 教訓：**知識カットオフ（2026年1月）以降のモデル・プラン情報は思い込みで答えず、必ずWeb検索で確認する**。

### Proプランで今回のゲームは作れるか？
| 観点 | 見立て |
|---|---|
| モデル品質 | **Proでも同じFable 5が使える**（6/22まで無料枠）。品質差はなし |
| 5時間枠 | ここが唯一の実質的制約。v1規模（実質130万トークン級）は1枠に収まらず、**2〜4枠に分割**が必要 |
| 週間上限 | v1だけなら収まる。v2＋オンライン対戦まで全部だと週上限も意識 |

### 結論（人に説明する用）
> **「Proでも同じ品質のゲームが作れます。モデルも同じものが使えます（6/22までは無料提供枠）。違いはレート制限のペース配分だけ。Proなら2〜3セッションに分けてじっくり、Maxなら1日で一気に、という体験の差です」**

- まずProで試して、開発が続きそうならMaxに上げる、という順番が無駄がない。
- **6/22まで全プランでFable 5が無料**なので、試すなら今が一番お得。

## 7. なぜ今回トークン効率が良かったのか（再現可能なコツ）

1. **サブエージェント分業**
   - 重い作業（キャラ制作1体9〜11万トークン）を各エージェントの独立コンテキストで消費。
   - メインの会話は「仕様書を渡す→結果サマリを受け取る」だけなので、メイン側の文脈が膨らまない。
   - ※ただし並列数を増やすとquotaを一気に食う。Proなら**並列数を絞る**（4体同時→2体ずつ）のが安全。
2. **アセットを全部テキスト/コードにした**
   - 画像生成を一切使わず、ドット絵もBGMもステージもコード。生成→確認→修正のループが軽い。
3. **検証の自動化**
   - validate.js や Playwright テストが機械的に合否を返すので、試行錯誤の回数自体が少ない。
4. **仕様書ファースト**
   - データ契約を先に書いてからエージェントに依頼 → 手戻りが減る（＝無駄なトークンを使わない）。

## 8. トークン実測のやり方（再利用メモ）

トランスクリプトは以下にある：
```
C:\Users\intel\.claude\projects\<プロジェクトのスラッグ>\<セッションID>.jsonl
```
- 各行が1イベントのJSON。`type: "assistant"` の行の `message.usage` にトークン数が入っている。
- フィールド: `input_tokens` / `output_tokens` / `cache_read_input_tokens` / `cache_creation_input_tokens`
- サブエージェントの消費は、親トランスクリプトの完了通知に `subagent_tokens` として載る（メインループの集計には含まれない点に注意）。
- 集計スクリプトは `dot-damashii/tools/token_report.js` に保存済み（マーカー文字列でフェーズ分割できる）。
- リアルタイムの残量は `/usage` コマンド、または設定したステータスラインで確認可能。

- **実質コストは「出力トークン（コード生成）」が主役。キャッシュ読出は見た目デカいが激安。**

- **重い開発と軽い質問はセッションを分ける。一段落したら `/compact`。**
